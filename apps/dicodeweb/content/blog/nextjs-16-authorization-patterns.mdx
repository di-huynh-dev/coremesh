When an App Router codebase is still small, it is tempting to treat route guarding as a single problem and solve all of it in one place.

That is where teams start overloading `middleware.ts`, or in modern Next.js 16 terms, `proxy.ts`.

The trouble is that authorization is not one decision. It is a stack of decisions:

- should this request continue at all?
- should this user see this screen?
- should this mutation be allowed?
- should this user access this specific record?

If one layer tries to answer all four questions, the system becomes slow, brittle, or both.

The scalable mental model is simpler:

- `proxy.ts` handles fast, optimistic request gating
- the data access layer handles secure session validation
- Server Components, Route Handlers, and Server Actions enforce the final authorization rules near the data

## Start with the Next.js 16 Shift

One important update is easy to miss if you learned this area from older tutorials: in Next.js 16, the `middleware` file convention was renamed to `proxy`.

That naming change matters because it clarifies intent. The official docs position Proxy as a network boundary that runs before a route is rendered, and recommend avoiding it unless it is actually the right tool for the job.

That is a useful constraint.

Proxy is great at cheap decisions. It is not where your entire authorization model should live.

<Callout type="tip" title="A Better Rule Of Thumb">
  Let Proxy answer <strong>"can this request continue?"</strong>. Let your server-side data layer
  answer <strong>"what may this user actually access or change?"</strong>.
</Callout>

## Pattern 1: Keep Proxy Small and Cheap

The official authentication guide recommends using Proxy for optimistic checks and explicitly warns against doing database work there on every matched request.

That recommendation becomes more important as an app grows, because Proxy runs before rendering and can also run on prefetched routes. If it becomes heavy, the whole app feels heavy.

This is the kind of Proxy that scales well:

```ts
import { NextRequest, NextResponse } from 'next/server';

const publicPaths = ['/', '/login', '/signup'];

function isPublicPath(pathname: string) {
  return publicPaths.some((path) => pathname === path || pathname.startsWith(`${path}/`));
}

export default function proxy(request: NextRequest) {
  const { pathname } = request.nextUrl;
  const hasSession = Boolean(request.cookies.get('session')?.value);

  if (!hasSession && !isPublicPath(pathname)) {
    return NextResponse.redirect(new URL('/login', request.url));
  }

  if (hasSession && (pathname === '/login' || pathname === '/signup')) {
    return NextResponse.redirect(new URL('/dashboard', request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: ['/((?!api|_next/static|_next/image|.*\\.(png|jpg|jpeg|svg|webp|ico)$).*)'],
};
```

Why this works:

- it checks session presence, not deep permissions
- it keeps the matcher focused
- it avoids putting database traffic on every matched request
- it turns Proxy into a perimeter guard instead of a policy engine

## Pattern 2: Put Secure Authorization in a DAL

The official Next.js auth guide recommends creating a Data Access Layer and centralizing authorization there.

That is the real foundation.

Your DAL is where you verify the session in a way that can safely touch databases, session stores, role tables, team membership, resource ownership, or revocation state.

```ts
import 'server-only';

import { cache } from 'react';
import { cookies } from 'next/headers';
import { redirect } from 'next/navigation';
import { db } from '@/lib/db';

export const verifySession = cache(async () => {
  const token = (await cookies()).get('session')?.value;

  if (!token) {
    redirect('/login');
  }

  const session = await db.sessions.findValidByToken(token);

  if (!session) {
    redirect('/login');
  }

  return {
    userId: session.userId,
    role: session.role,
    teamId: session.teamId,
  };
});
```

Then every sensitive read or write builds on that verified session instead of trusting navigation-level gating.

```ts
export async function getProject(projectId: string) {
  const session = await verifySession();

  const project = await db.projects.findById(projectId);

  if (!project || project.teamId !== session.teamId) {
    return null;
  }

  return project;
}
```

This is the difference between “we redirected the browser” and “we actually enforced access control.”

## Pattern 3: Use Layouts and Pages for UI Decisions, Not Final Security

Layouts are good at shaping user experience.

They are not enough by themselves to secure the system.

For example, a dashboard layout can fetch the current user, select a nav variant, or redirect an unauthenticated visitor early. That is helpful. But if the same user can still hit a Route Handler or trigger a Server Action that does not re-check permissions, the system is still incomplete.

This is where many apps start to feel flaky. The UI looks protected, but the server contract is weaker than the shell around it.

## The Anti-Patterns That Age Badly

There are a few patterns that look reasonable in week one and become painful in month six.

### Doing database or revocation checks in Proxy

This is the first place teams accidentally turn every navigation into backend infrastructure.

If Proxy is checking a session table, a permissions table, and a tenant membership table on every matched request, you have made the fastest part of routing depend on the slowest part of your system.

The official docs recommend avoiding database checks in Proxy for exactly this reason.

### Treating Proxy as the only guard for APIs

Perimeter checks matter. They are not sufficient.

If a Route Handler performs sensitive work, it should verify session and permissions itself. The same applies to Server Actions. Internal calls and new code paths accumulate over time; assuming every access path is safely covered by a perimeter layer is how silent holes appear.

### Encoding too much role logic into redirects

Aggressive redirect logic can create the kind of bugs that feel like race conditions even when the underlying issue is architectural.

Why? Because App Router navigation includes prefetching, nested layouts, and server-rendered boundaries. When Proxy keeps issuing role-based redirects for every nuance of authorization, the navigation layer starts carrying decisions that really belong to your secure server layer.

That is where you see symptoms such as:

- content flashes before a redirect settles
- redirect loops between public and private routes
- behavior that differs between hard refresh and client navigation

A safer rule is this:

- use Proxy redirects for missing or obviously invalid sessions
- use server-side data checks for permissions, ownership, and resource-specific access
- use `redirect()` or `notFound()` from the server layer once you know the real business decision

### Matching too broadly

If your matcher includes assets, images, or routes that do not need guarding, you pay for work that produces no security value.

Even a cheap Proxy becomes expensive when it runs unnecessarily across the whole request surface.

## A Practical Architecture For Real Apps

If I were setting up authorization in a fresh Next.js 16 App Router project today, I would use this split:

<Steps>
  <Step number="01" title="Proxy for perimeter control">
    Check whether a session cookie exists, keep the matcher narrow, and redirect obvious
    unauthenticated traffic away from protected routes.
  </Step>
  <Step number="02" title="DAL for secure session verification">
    Centralize session validation, role lookups, team membership, and ownership checks in
    server-only code close to the database.
  </Step>
  <Step number="03" title="Layouts and pages for experience">
    Use server-rendered layouts to branch the interface, show the right shell, and improve the feel
    of protected navigation without making them your only security layer.
  </Step>
  <Step number="04" title="Handlers and actions for final enforcement">
    Every sensitive mutation and every sensitive resource read should verify session and
    authorization again where the work actually happens.
  </Step>
</Steps>

This structure is slightly more repetitive than the “do it all in middleware” tutorial version.

It is also much easier to audit, test, and evolve.

## The Core Idea To Keep

The biggest mistake is not choosing the wrong helper function. It is assigning the wrong responsibility to the wrong layer.

Proxy is excellent when you keep it fast and boring.

Authorization becomes robust when the final decision lives next to the data that decision protects.

That is the version that scales: quick perimeter checks up front, secure checks deeper in the stack, and no illusion that redirects alone are the same thing as access control.
